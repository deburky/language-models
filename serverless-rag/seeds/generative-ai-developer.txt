## Model types and selection

 BioGPT for biomedical text generation, AlphaFold for protein structure prediction, LEGAL-BERT for legal document analysis, and CodeT5 for software development tasks.

1. **Choose general-purpose models when:** You need to perform many different kinds of tasks across various domains. Benefit: Offers flexibility and breadth across multiple applications, reducing the need for multiple specialized models. Cost-effective for diverse workloads.
2. **Choose specialized models when:** You need to perform a single, critical task that requires high precision in a specific domain. ## Integration approaches

> [!important]
> **Amazon Bedrock integration (unified API):**
> Amazon Bedrock provides a unified API to access multiple foundation models without managing infrastructure. This fully managed service offers pay-per-use pricing based on API calls, with no infrastructure management required. Benefits include quick time-to-market, built-in security and compliance, access to various models through a single interface, and automatic scaling. Best for applications without deep ML expertise, rapid prototyping, standard use cases, and when you want to focus on application logic rather than infrastructure.
> 
> **Amazon SageMaker AI integration (self-host):**
> Amazon SageMaker AI enables custom model deployment and training with full control over model configuration. This option charges based on compute resources and storage, giving you complete control over deployment settings, instance types, and scaling policies. Benefits include ability to deploy custom models, extensive fine-tuning capabilities, specialized configurations for compliance, and integration with existing ML pipelines. Best for custom models, applications requiring extensive fine-tuning, specialized requirements, or when you need granular control over the ML infrastructure.
> 
> **Direct provider API integration (direct API):**
> Direct provider APIs involve integrating with model providers like Anthropic or OpenAI through their native APIs or frameworks like LangChain. This approach provides direct access to the latest model features and updates as soon as providers release them. Benefits include access to provider-specific capabilities, flexibility in integration patterns, and potential for early access to new features. Consider for applications with specific provider requirements, when you need features not yet available in Bedrock, or when direct provider relationships offer advantages.

**Fine-tuning solution**: fine-tuing adjusts model parameters using your specific training data to improve performance for domain-specific tasks. Requires training and validation datasets, and typically needs **Provisioned Throughput** for deployment.

**The RAG solution:** RAG combines foundation models with external knowledge sources to provide more accurate, up-to-date, and contextually relevant responses. When a user asks a question, RAG retrieves relevant information from your knowledge base and provides it as context to the foundation model, grounding the response in factual sources.

_Model chaining_ involves connecting multiple AI models or services to create complex workflows. _Sequential chaining_ provides linear flow from one model to another where output becomes input to the next. _Parallel processing_ uses multiple models processing same input simultaneously with results combined for final output. _Conditional branching_ uses different models based on input characteristics, whereas _feedback loops_ use model outputs to influence subsequent processing steps.

> [!example] Document processing pipeline uses OCR for text extraction, then classification, summarization, and storage with each step optimized for specific task. Multi-modal content generation flows from text prompt to image generation to image analysis to caption generation by using different models for different modalities. Intelligent customer service implements intent classification, knowledge retrieval, response generation, and sentiment analysis with adaptive routing based on complexity.

**Converse API**

```python
import boto3

bedrock = boto3.client('bedrock-runtime')

def invoke_model(prompt, model_id="anthropic.claude-3-5-sonnet-20240620-v1:0"):
	response = bedrock.converse(
		modelId=model_id,
	    messages=[{"role": "user", "content": [{"text": prompt}]}],
	    inferenceConfig={"maxTokens": 1000}
	)
  return response['output']['message']['content'][0]['text']
```

Batch processing

```python
import time
from concurrent.futures import ThreadPoolExecutor

def process_batch(items, max_workers=5):
    def process_item(item):
        try:
            return invoke_model(item["prompt"])

        except Exception as e:
            time.sleep(1)  # Simple retry delay

            return invoke_model(item["prompt"])

    with ThreadPoolExecutor(max_workers=max_workers) as executor:
        results = list(executor.map(process_item, items))

    return results
```

### Dynamic Model Routing

> Dynamic model routing makes **foundation model selection** decisions at runtime based on real-time analysis of input characteristics, system performance, and business requirements.

![[bedrock_dynamic_routing.png]]

> *Amazon Bedrock Intelligent Prompt Routing*.
#### API Gateway Routing

This Lambda function dynamically selects the optimal Claude model based on user tier and content length. It analyzes incoming requests, routing premium users with long content (>1000 characters) to Claude 3 Opus, medium-length content (>500 characters) to Claude 3 Sonnet, and shorter content to Claude 3 Haiku.

```python
import json
import boto3

def lambda_handler(event, context):
    bedrock = boto3.client('bedrock-runtime')
    
    # Analyze request characteristics
    content_length = len(event['body'])
    user_tier = event['headers'].get('X-User-Tier', 'basic')
    
    # Dynamic model selection logic
    if user_tier == 'premium' and content_length > 1000:
        model_id = 'anthropic.claude-3-opus-20240229-v1:0'
    elif content_length > 500:
        model_id = 'anthropic.claude-3-sonnet-20240229-v1:0'
    else:
        model_id = 'anthropic.claude-3-haiku-20240307-v1:0'
    
    # Invoke selected model
    response = bedrock.invoke_model(
        modelId=model_id,
        body=json.dumps({
            "anthropic_version": "bedrock-2023-05-31",
            "max_tokens": 1000,
            "messages": [{"role": "user", "content": event['body']}]
        })
    )
    
    return {
        'statusCode': 200,
        'body': response['body'].read(),
        'headers': {
            'Content-Type': 'application/json',
            'X-Selected-Model': model_id
        }
    }
```

## Circuit breaker patterns

Circuit breaker patterns **protect distributed systems from cascading failures** by monitoring service health and <mark style="background-color: tomato; color: white;"><u>automatically stopping requests to failing services</u></mark>. In AI applications, circuit breakers prevent failed model calls from overwhelming downstream systems and provide graceful degradation when services become unavailable.

![[bedrock_circuit_breakers.png]]

> [!success] Three states:
> * Closed (normal operation)
> * Open (blocking requests)
> * Half-open (testing recovery)

Circuit breaker patterns provide the following **benefits** for AI applications:

- **Failure isolation**: Prevent failing AI services from affecting other system components.
- **Resource protection**: Avoid wasting resources on requests likely to fail.
- **Automatic recovery**: Resume normal operation when services recover.
- **Graceful degradation**: Maintain partial functionality during service disruptions.

> The purpose of circuit breaker patterns in AI applications is to prevent cascading failures and provide graceful degradation.

**Cross-region failover**: Implement cross-region failover logic that routes requests to AI services in different AWS Regions when regional services become unavailable.

Cross-region inference region selection:
* Geographic proximity to users and applications
* The number of available foundation models in each region

> [!success] [!success] **Best practicess*:
> Implement sliding window calculations to track recent error rates and avoid false positives from temporary service fluctuations.
> Imlemenent degraded functionality with cached responses or rule-based systems to maintain core functionality.
> Use Amazon DynamoDB or other storage services to track circuit states, error counts, and timing information.

### Graceful Degradation

Graceful degradation (*a design principle*) maintains essential functionality when AI systems experience disruptions or failures. It covers strategies for implementing fallback mechanisms that preserve core capabilities, manage user expectations, and maintain service continuity during various types of system disruptions.
### Cached Responses

Serve previously generated results from cache when no other options are available. This approach maintains some level of functionality by reusing past AI outputs for similar requests. Configure intelligent caching that can match current requests to relevant cached responses based on similarity or context.

1. Intelligent caching (serve cached responses for requests that closely match previous queries)
2. Precomputed responses (batch-generate responses)
3. Design cache management systems that prioritize high-value responses and adjust retention policies based on system health.
4. During disruptions, extend cache retention to maximize available responses.

```python
def truncate_conversation(messages, max_tokens, tokenizer):
    total_tokens = 0
    truncated_messages = []

    # Always keep system message
    if messages[0]["role"] == "system":
        truncated_messages.append(messages[0])
        total_tokens += count_tokens(messages[0]["content"], tokenizer)
        messages = messages[1:]

    # Add messages from most recent, working backwards
    for message in reversed(messages):
        message_tokens = count_tokens(message["content"], tokenizer)

        if total_tokens + message_tokens <= max_tokens:
            truncated_messages.insert(-1 if truncated_messages else 0, message)
            total_tokens += message_tokens
        else:
            break

    return truncated_messages
```

> **Cosine similarity** is calculated by using the dot product of normalized vectors. This approach makes cosine similarity particularly effective for comparing documents of different lengths or when magnitude differences don't indicate semantic differences.

S3 Vector Store example:

```python
import boto3
import json
import numpy as np

class S3VectorStore:
    def __init__(self, bucket_name='vector-store', prefix='vectors/'):
        self.s3 = boto3.client('s3')
        self.bucket_name = bucket_name
        self.prefix = prefix
    
    def store_vector(self, doc_id, embedding, metadata=None):
        """Store a vector embedding with metadata in S3"""
        if metadata is None:
            metadata = {}
        
        # Convert numpy arrays to lists for JSON serialization
        if isinstance(embedding, np.ndarray):
            embedding = embedding.tolist()
        
        # This is the core operation - storing the vector in S3
        self.s3.put_object(
            Bucket=self.bucket_name,
            Key=f'{self.prefix}{doc_id}.json',
            Body=json.dumps({'vector': embedding, 'metadata': metadata}),
            ContentType='application/json'
        )
        return {'status': 'success', 'doc_id': doc_id}
    
    def get_vector(self, doc_id):
        """Retrieve a vector by document ID"""
        response = self.s3.get_object(
            Bucket=self.bucket_name,
            Key=f'{self.prefix}{doc_id}.json'
        )
        data = json.loads(response['Body'].read().decode('utf-8'))
        return data
    
    def search_similar(self, query_vector, top_k=5):
        """Find similar vectors using cosine similarity"""
        # List all vectors
        response = self.s3.list_objects_v2(
            Bucket=self.bucket_name,
            Prefix=self.prefix
        )
        results = []
        
        # Process each vector
        for item in response.get('Contents', []):
            # Get vector data
            vector_data = self.get_vector(
                item['Key'].replace(self.prefix, '').replace('.json', '')
            )
            
            # Calculate similarity
            similarity = self._cosine_similarity(
                query_vector,
                vector_data['vector']
            )
            
            results.append({
                'doc_id': item['Key'].replace(self.prefix, '').replace('.json', ''),
                'similarity': similarity,
                'metadata': vector_data['metadata']
            })
        
        # Sort by similarity (highest first)
        results.sort(key=lambda x: x['similarity'], reverse=True)
        
        # Return top k results
        return results[:top_k]
    
    def _cosine_similarity(self, v1, v2):
        """Calculate cosine similarity between two vectors"""
        v1 = np.array(v1)
        v2 = np.array(v2)
        return np.dot(v1, v2) / (np.linalg.norm(v1) * np.linalg.norm(v2))
```

```python
def evaluate_reranker_model(bedrock_client, model_id, query_document_pairs):
    """Evaluate Bedrock reranker model performance"""
    results = []
    
    for query, documents in query_document_pairs:
        # Prepare reranking request
        rerank_request = {
            'queries': [{'type': 'TEXT', 'textQuery': {'text': query}}],
            'sources': [
                {
                    'type': 'INLINE',
                    'inlineDocumentSource': {
                        'type': 'TEXT',
                        'textDocument': {'text': doc['content']}
                    }
                }
                for doc in documents
            ],
            'rerankingConfiguration': {
                'type': 'BEDROCK',
                'bedrockRerankingConfiguration': {
                    'modelConfiguration': {
                        'modelArn': f'arn:aws:bedrock:region::foundation-model/{model_id}'
                    },
                    'numberOfResults': len(documents)
                }
            }
        }
        
        # Run the reranking
        response = bedrock_client.rerank(**rerank_request)
        
        # Process results
        reranked_results = [
            {
                'document_id': result['index'],
                'relevance_score': result['relevance_score'],
                'original_rank': idx
            }
            for idx, result in enumerate(response['results'])
        ]
        
        results.append({
            'query': query,
            'reranked_results': reranked_results
        })
    
    return results
```

## RAG Caching

Effective RAG caching implementations leverage three complementary approaches to optimize performance while maintaining information freshness:

- **Tiered caching architecture:** Implements multi-level caching across memory, disk, and distributed systems to optimize retrieval based on access patterns. Supports both query-based and document-based approaches, with semantic similarity caching extending utility to conceptually related queries.
- **Adaptive caching mechanisms:** Dynamically adjusts caching behavior using frequency analysis, context awareness, and predictive prefetching. Allocates resources efficiently by prioritizing commonly accessed information and incorporating user session data for relevance.
- **Embedding-based caching:** Stores vector embeddings rather than full documents, significantly reducing storage requirements while maintaining semantic search capabilities. Utilizes approximate nearest neighbor lookups and compression techniques to optimize memory utilization.

### RAG API

The *RetrieveAndGenerate* API utilizes two important parameters as follows:

- **numberOfResults**: The *numberOfResults* parameter in the given function determines the number of search results that are retrieved from the knowledge base and included in the prompt provided to the model for generating an answer. Specifically, it fetches the top *max_results* number of documents or search results that most closely match the given query.

- **textPromptTemplate**: The *textPromptTemplate* parameter is a string that serves as a template for the prompt that is provided to the model. In this case, the *default_prompt* is being used as the template. This template includes placeholders (`$search_results$` and `$output_format_instructions$`) that are replaced with the actual search results and any output format instructions, respectively, before being passed to the model.

Strands Agents provides three primary multi-agent patterns: Graph, Swarm, and Workflow. While all three patterns enable agent collaboration, they differ fundamentally in how execution paths are determined and controlled. Understanding these differences helps you select the pattern that best fits your use case.

Graph pattern is ideal for business processes that require branching logic, loops, or explicit error handling.

Unlike Graph, you don't define the execution path for Swarm. Instead, you provide a pool of specialized agents, and they decide among themselves who should handle each task. Agents share memory and context: this exemplifies emergent intelligence where the collective capabilities exceed what any single agent could achieve.

Unlike Graph and Swarm, the Workflow pattern doesn't allow cycles and follows a completely deterministic execution path. Workflow pattern is ideal for repeatable, complex operations that require reliability and predictability.

## Introduction to Model Context Protocol

Model Context Protocol is a framework that facilitates interactions between agents and tools, providing structured approaches to managing context and state across different interactions. MCP addresses the challenges of maintaining coherent communication and state management in complex multi-agent environments.

# 8. Model Deployment Strategies

AWS provides several deployment strategies to address the challenges of deploying generative AI models.

- You can use AWS Lambda for on-demand, event-driven workloads that require low latency.
- Amazon Bedrock offers provisioned throughput configurations for consistent performance in high-throughput scenarios. (Amazon Bedrock also supports on-demand inference)
- Amazon SageMaker AI endpoints provide hybrid solutions that combine on-demand and provisioned capabilities for applications with varying workloads.

Amazon Bedrock provisioned throughput provides consistent, predictable performance for high-throughput, steady-state workloads that require **guaranteed capacity** and minimal latency variation. This approach pre-allocates dedicated resources to maintain consistent performance characteristics.

Amazon SageMaker AI endpoints provide hybrid solutions that combine **on-demand and provisioned capabilities**, offering flexible deployment options that adapt to varying workloads through intelligent auto-scaling and resource management.

> [!attention] Considerations
> Lambda functions provide cost efficiency for variable workloads through pay-per-use pricing but may become expensive for consistent high-volume usage.
> 
> Provisioned throughput offers predictable costs for steady-state workloads but requires payment for reserved capacity regardless of actual utilization.
> 
> SageMaker endpoints provide cost flexibility through auto-scaling but require careful configuration to avoid over-provisioning during low-usage periods.

## AWS Lambda Functions for Timeout Mechanisms

AWS Lambda functions provide built-in timeout capabilities that help you manage resource consumption and prevent functions from running indefinitely.

For long-running tasks, implement **asynchronous processing patterns** where Lambda functions initiate tasks and use Amazon SQS or Amazon SNS to handle results. This approach prevents timeout issues although it maintains workflow continuity and system responsiveness.

```python
# Example: Implementing asynchronous processing with SQS
import boto3
import json
import uuid

sqs = boto3.client('sqs')
lambda_client = boto3.client('lambda')

def initiator_lambda_handler(event, context):
    # This function initiates a long-running process
    job_id = str(uuid.uuid4())

    # Break the large task into smaller chunks
    chunks = split_large_task(event['data'])

    # Send each chunk to SQS for processing
    queue_url = 'https://sqs.us-east-1.amazonaws.com/123456789012/ProcessingQueue'

    for i, chunk in enumerate(chunks):
        message = {
            'job_id': job_id,
            'chunk_id': i,
            'total_chunks': len(chunks),
            'data': chunk
        }

        # Send to SQS queue
        sqs.send_message(
            QueueUrl=queue_url,
            MessageBody=json.dumps(message)
        )

    return {
        'status': 'initiated',
        'job_id': job_id,
        'chunks_queued': len(chunks)
    }
# In a separate Lambda function triggered by SQS
def worker_lambda_handler(event, context):
    for record in event['Records']:
        message = json.loads(record['body'])

        # Process the chunk
        result = process_chunk(message['data'])

        # Store results in DynamoDB or S3
        store_result(message['job_id'], message['chunk_id'], result)

        # If this is the last chunk, trigger completion Lambda
        if is_last_chunk(message['job_id'], message['chunk_id'], message['total_chunks']):
            lambda_client.invoke(
                FunctionName='completion-function',
                InvocationType='Event',
                Payload=json.dumps({'job_id': message['job_id']})
            )
```
